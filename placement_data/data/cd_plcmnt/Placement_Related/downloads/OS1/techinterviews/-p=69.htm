    Good C++ Interview questions

    Some good C++ questions to ask a job applicant.

    1. How do you decide which integer type to use?
    2. What should the 64-bit integer type on new, 64-bit machines be?

    3. What’s the best way to declare and define global variables?
    4. What does extern mean in a function declaration?
    5. What’s the auto keyword good for?
    6. I can’t seem to define a linked list node which contains a pointer to itself.
    7. How do I declare an array of N pointers to functions returning pointers to functions returning pointers to characters?
    8. How can I declare a function that returns a pointer to a function of its own type?
    9. My compiler is complaining about an invalid redeclaration of a function, but I only define it once and call it once. What’s happening?
    10. What can I safely assume about the initial values of variables which are not explicitly initialized?
    11. Why can’t I initialize a local array with a string?
    12. What is the difference between char a[] = “string”; and char *p = “string”; ?
    13. How do I initialize a pointer to a function?

    8. How do I declare an array of N pointers to functions returning pointers to functions returning pointers to characters?

      If you want the code to be even slightly readable, you will use typedefs.

      typedef char* (*functiontype_one)(void);
      typedef functiontype_one (*functiontype_two)(void);
      functiontype_two myarray[N]; //assuming N is a const integral

      What’s the auto keyword good for?
      Not much. It declares an object with automatic storage duration. Which means the object will be destroyed at the end of the objects scope. All variables in functions that are not declared as static and not dynamically allocated have automatic storage duration by default. For example

      int main()
      {
      int a; //this is the same as writing “auto int a;”
      }

    11. In response to Sachin/Russ’s answers:
      what int type to use:
      C/C++ are very loose in interpretation of what ‘int’ is. Use the one that gives you enough space, considering your platform and compiler. If you don’t know for sure that this will always be constant, typedef it so that you can change it later. Good example of this is porting code for 64 bit computers from 32.
      C/C++ define int to be something that holds an integer value. Short int is guaranteed to not be larger than int, and long int is guaranteed to not be smaller. Both can be the same as int. It is up to the implemntation as to what int is. On 32-bit machines, this is often (for speed’s sake) implemented such that ‘int’ is 32 bit, short int is smaller, and is 16bits, long int is 32bits (since in 16-bit days ’short’ used to be the same as regular int, and long was 32 bit). Compilers that support it have ‘long long’ which is 64-bit (gcc supports it. Visual C++ has __int64). On 64-bit architectures, int would be 64-bit, long int would be at least that, and short int probably becomes 32bits. How to create a 16-bit value then? I don’t know :)

      extern:
      Gives information to the compiler about something that will be provided at a later point from another source module (most likely during the static link at the end of the compile phase). Also able to change the calling convention for functions (and thus C++ programs use extern “C” {} to signify that the functions in there export as C style calling conventions.) What I think sachin is trying to say in his clarification is that you can have both an extern and the normal definition in the same file, and it won’t cause problems. But the real definition needs to be made somewhere, and in only one spot. Whereas the declaration (which contains the extern) needs to be made everywhere before it is used.

      global variables:
      How global do you want them? Provide an ‘extern’ declaration in a header file. include that inside of whatever files need access to this variable. In a source file, have the same definition without the extern: this is what actually allocates the memory inside the executable. Without this, you basically have an unfulfilled promise that there will be a variable with this name linked in, but all the source modules expect it to come from somewhere else. Duplicating the definition without the export line is important. This will make the variable global to any part of the project that includes the header file with the extern definition. If you include it in a header file without the extern definition, and have that header file used more than once in the entire project, you’ll get collisions at link time (since multiple files claim they own the global variable foo).

      Global to the file only (this is only a scoping issue.. it is still technically a global variable, but is hidden from all other source files by the compiler), make the variable static. This will make a global variable only accessible by that source file (and any #included after its definition, I believe. I’m not sure on this. That’s bad form anyway). So multiple source files can have a global variable that is local to that source file, with the same name. This does not cause issues at link time. But, they don’t share space or share the value, so while it is still technically global (all functions in that source file have access to it, and it acts like a global variable), it is not accessible to any other source files.

      Hopefully that clears up some confusion :)
